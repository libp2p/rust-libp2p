# Protobuf Memory Amplification Problem Specification
## Executive Summary
This document describes a fundamental vulnerability in protobuf decoding within the rust-libp2p ecosystem. A malicious remote peer can exploit the memory amplification that occurs when decoding small protobuf messages into larger Rust structures, causing memory exhaustion and potential denial-of-service (DoS) attacks.
**Impact:** A single 12 MB gossipsub RPC frame can allocate ~980 MB of physical memory (82× amplification) before any application-level validation is performed.
## Problem Description
### Core Issue
The prost crate (and protobuf libraries in general) decode wire-format messages into fully-initialized Rust structs. The amplification occurs because:
1. **Wire format is compact**: Protobuf uses variable-length encoding where optional fields can be omitted
2. **Rust structures are verbose**: Each field requires memory for discriminant (Option), heap allocation metadata (Vec/String), and alignment padding
3. **No pre-validation**: The decoder has no way to limit the number of repeated elements before allocating memory for them
### Memory Amplification Math
#### Minimal Gossipsub Publish Message Example
A minimal publish message in wire format requires only **4 bytes**:
```
0x12  field 2 (publish), wire type 2 (length-delimited)
0x02  inner message length = 2
0x22  field 4 (topic), wire type 2 (length-delimited)
0x00  topic length = 0 (empty string)
```
#### Memory Footprint Breakdown
**proto::Message (generated by prost)**
```rust
#[derive(Clone, PartialEq, Eq, Hash, ::prost::Message)]
pub struct Message {
    #[prost(bytes = "vec", optional, tag = "1")]
    pub from: ::core::option::Option<::prost::alloc::vec::Vec<u8>>,     // ~24 bytes
    #[prost(bytes = "vec", optional, tag = "2")]
    pub data: ::core::option::Option<::prost::alloc::vec::Vec<u8>>,     // ~24 bytes
    #[prost(bytes = "vec", optional, tag = "3")]
    pub seqno: ::core::option::Option<::prost::alloc::vec::Vec<u8>>,    // ~24 bytes
    #[prost(string, required, tag = "4")]
    pub topic: ::prost::alloc::string::String,                           // ~24 bytes
    #[prost(bytes = "vec", optional, tag = "5")]
    pub signature: ::core::option::Option<::prost::alloc::vec::Vec<u8>>,// ~24 bytes
    #[prost(bytes = "vec", optional, tag = "6")]
    pub key: ::core::option::Option<::prost::alloc::vec::Vec<u8>>,      // ~24 bytes
}
```
Estimated size: **~144 bytes** (6 × 24 bytes per field + struct overhead)
**RawMessage (application type)**
```rust
pub struct RawMessage {
    pub source: Option<PeerId>,           // ~40+ bytes (PeerId contains multihash)
    pub data: Vec<u8>,                    // ~24 bytes
    pub sequence_number: Option<u64>,     // ~16 bytes
    pub topic: TopicHash,                 // ~24+ bytes (contains String)
    pub signature: Option<Vec<u8>>,       // ~24 bytes
    pub key: Option<Vec<u8>>,             // ~24 bytes
    pub validated: bool,                  // ~1 byte + padding
}
```
Estimated size: **~112+ bytes**
#### Amplification Calculation
With `max_transmit_size = 12,234,442` bytes:
- **Messages per frame:** 12,234,442 / 4 = **3,058,610 messages**
- **proto::Message allocation:** 3,058,610 × 144 bytes = **~440 MB**
- **RawMessage allocation:** 3,058,610 × 112 bytes = **~343 MB**
- **Total allocation:** **~783 MB minimum** (before heap fragmentation and allocator overhead)
- **Measured amplification:** **~82×** (980 MB / 12 MB)
## Attack Vector
### Exploitation Requirements
1. **No authentication required:** Attacker completes Noise XX handshake with self-generated keypair
2. **No rate limiting:** Single frame triggers synchronous allocation
3. **Minimal payload:** 4 bytes per message, empty topic passes `ValidationMode::Anonymous`
4. **Network agnostic:** Works on any network (holesky, mainnet, etc.)
### Attack Flow
```
1. Open TCP connection to target libp2p port (default 9000)
2. Complete Noise XX handshake
3. Negotiate /meshsub/1.1.0 via multistream-select
4. Send 12 MB protobuf frame with 3M+ minimal publish messages
5. Victim's GossipsubCodec::decode() allocates ~980 MB
6. On memory-constrained nodes (≤2 GB), OOM killer terminates process
```
### Common Pattern
All affected protocols follow this pattern:
```rust
impl Decoder for SomeCodec {
    type Item = SomeRpc;

    fn decode(&mut self, src: &mut BytesMut) -> Result<Option<Self::Item>, Self::Error> {
        // PROBLEM: This decodes ALL repeated elements into memory
        let rpc = prost_codec::Codec::decode(src)?;

        // Limits applied AFTER full decode - too late!
        if rpc.messages.len() > MAX_MESSAGES {
            rpc.messages.truncate(MAX_MESSAGES);
        }

        // More allocations happen here
        let mut messages = Vec::with_capacity(rpc.messages.len());
        // ...
    }
}
```
## Desired Properties of a Solution
### Security Requirements
1. **Fail-fast:** Reject excessive RPCs before significant memory allocation
2. **Resource bounds:** Guarantee maximum memory usage per RPC
3. **Backward compatibility:** Valid RPCs within limits must continue to work
4. **Observability:** Log when limits are hit for monitoring
### Performance Requirements
1. **Minimal overhead:** Don't significantly slow down normal operations
2. **Lazy evaluation:** Don't parse fields that won't be used
3. **Streaming support:** Process large RPCs without full materialization (future enhancement)

## Solution Approaches
### Option A: Pre-decode Count Check
Parse protobuf wire format to count repeated elements without full decoding.

**Pros:**
- Minimal memory allocation
- Fast rejection of malicious payloads
- No changes to prost

**Cons:**
- Complex to implement correctly (need to parse varints, wire types)
- Duplicates protobuf parsing logic
- Must stay in sync with protobuf schema changes
### Option B: Streaming/Incremental Decoding
Decode protobuf incrementally using a streaming parser.

**Pros:**
- Theoretically optimal memory usage
- Natural fit for async Rust
**Cons:**
- prost doesn't support streaming
- Would require switching to a different protobuf library
- Major refactoring of all protocols
### Option C: Custom prost Integration
Modify prost-codec to accept limits on repeated field counts.

**Pros:**
- Integrated solution
- Can fail during decode
- Minimal API changes

**Cons:**
- Requires modifying prost-codec or prost itself
- Complex to implement without prost support
### Option D: Bounded Collection Types
Use bounded collections that fail when capacity is exceeded.

**Pros:**
- Clean Rust implementation
- Can be generic

**Cons:**
- Still requires changes to generated prost code
- prost would need to support custom collection types

## Option E: RPC Size-Based Segmentation
Instead of limiting by message count, limit by RPC type with different size caps. This approach recognizes that different RPC types have vastly different amplification characteristics.
### Rationale
Analysis shows that different gossipsub RPC components have dramatically different memory amplification factors:
| RPC Component | Wire Bytes/Message | proto::Message Size | Amplification Factor |
|---------------|-------------------|---------------------|---------------------|
| **Publish Messages** | 4 bytes (minimal) | ~144 bytes | **36×** |
| **RawMessage** | 4 bytes wire | ~112 bytes | **28×** |
| **Control Messages** | 3-4 bytes | ~24-56 bytes | **6-14×** |
### Proposed Size Limits
| RPC Type | Size Limit | Rationale |
|----------|-----------|-----------|
| **Publish/Forward** | 12 MB | Allows large data payloads; requires additional protection |
| **Control RPCs** | 5 KB | Small control overhead; sufficient for normal operations |
| **Subscriptions** | 1 KB | Very small; only topic IDs |
### Amplification Analysis
**Publish RPC (12 MB limit):**
- Minimal message: 4 bytes (empty topic, no signature)
- Messages per frame: 12,234,442 / 4 = **3,058,610 messages**
- Memory allocation: 3,058,610 × ~144 bytes = **~440 MB** (proto::Message)
- RawMessage allocation: 3,058,610 × ~112 bytes = **~343 MB**
- **Total: ~783 MB from 12 MB wire** (65× amplification)
- **Status: HIGH RISK** - Requires additional protection
**Control RPC (5 KB limit):**
- Minimal ControlGraft: 4 bytes (empty topic_id)
- Messages per frame: 5,120 / 4 = **1,280 messages**
- Memory allocation: 1,280 × ~40 bytes = **~51 KB**
- **Total: ~51 KB from 5 KB wire** (10× amplification)
- **Status: LOW RISK** - Acceptable without additional protection
